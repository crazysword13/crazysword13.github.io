IDS实验

一、实验内容

二、实验环境

  • 靶机：Ubuntu，安装snort

（二）实验测试

1、Snort检测DoS（syn flood）攻击

Snort/rules目录下的local.rules文件中，设置如下规则：alert tcp any any -> $HOME_NET 80 (msg:"DoS attack"; flags:S; threshold:type threshold, track by_dst, count 100, seconds 60; sid:2000003)

注意要注释掉自带对应规则库。打开snort.conf，注释掉include $RULE_PATH/icmp-info.rules（608行）

之后启动snort，sudo snort -c /etc/snort/snort.conf -A fast -l /var/log/snort -i ens33

之后使用另一台kali使用hping3攻击（在dos实验中使用过的）：hping3 -S -p 80 --flood -V --rand-source 192.168.215.129

在日志中查看检测报告(cat /var/log/snort/alert)：

2、Snort检测nmap扫描

以nmap NULL扫描为例：

同理在local.rules中写下如下规则：alert tcp $EXTERNAL_NET any -> $HOME_NET any (msg:"Nmap Null PortScan!"; flags:0; threshold:type both,track by_dst,count 20,seconds 60; sid:10000012; rev:1;)

简单了解：Nmap隐蔽扫描有3个选项，分别为-sN、-sF、-sX。-sN是Null扫描，nmap NULL扫描是nmap秘密扫描的一种，相对比较隐蔽，通过向目标主机端口发送标志位flags为0的包进行扫描。通过发送非常规的TCP通信数据包对计算机进行探测,Null扫描不会标记任何数据包，目标主机相应端口关闭，会响应一个RST数据包，开发则不会响应任何信息

使用kali中的nmap对主机进行NULL扫描。namp -sN 192.168.215.129

查看snort报警信息。

3、Snort检测ssh连接

首先将两台机器（可以是主机和虚拟机）的ssh服务开启。

在local.rules中写入如下规则，检测ssh连接行为：alert tcp any any -> $HOME_NET 22 (msg:"MSF SSH brute force";content:"SSH-2.0";threshold:type threshold, track by_src, count 3, seconds 30; sid:2000004)

之后启动snort，sudo snort -c /etc/snort/snort.conf -A fast -l /var/log/snort -i ens33

但是尴尬的是，这里并没有查看到ssh的连接检测信息，通过查找网上的资料，在ssh - 使用snort/suricata，我想为登录到我的家庭网络的每次失败生成SSH警报 - IT工具网 (coder.work)中找到了原因，因为实际上我们是在尝试查看加密的内容(这将是身份验证和后续失败消息的位置)，但是采用它的方案，我们也并没有得到结果，只是查看到一堆无用的连接信息

之后启动snort，sudo snort -c /etc/snort/snort.conf -A fast -l /var/log/snort -i ens33

四、实验小结

通过本次实验，我学会了使用snort来编写一些规则，用于进行入侵检测，在本次实验中，我们尝试检测DoS攻击，namp扫描以及ssh连接，除了ssh连接无法检测，其余的入侵都成功的在日志中显示了信息，这个问题将在后续得到解决。总而言之，本次实验也是收获颇丰，学到了许多知识。

补充：通过查阅资料，最终解决了ssh连接的入侵信息检测，这让我对snort规则的编写有了更加深入和具体的了解了。
